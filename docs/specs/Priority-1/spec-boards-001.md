# spec-boards-001 — 보드 관리 PRD

## 1. 개요
- Modern Kanban Service 최초 온보딩에서 사용자가 프로젝트·팀 단위로 작업 공간을 생성하고 유지할 수 있도록 한다.
- 보드 생성·편집·삭제 흐름을 통합하여 “보드 라이프사이클”을 완성하는 것이 목표다.

## 2. 연계 요구사항
- FR-01 보드 생성
- FR-02 보드 편집
- FR-03 보드 삭제

## 3. 주요 사용자 시나리오
1. 팀원이 새 프로젝트 킥오프를 위해 보드를 등록하고 초기 컬럼/라벨 템플릿을 지정한다.
2. 프로젝트 리드가 진행 중인 보드의 이름·설명·색상 테마를 수정하여 조직 내 네이밍 규칙을 맞춘다.
3. 관리자 권한 사용자가 종료된 캠페인 보드를 삭제하고, 삭제 전 확인 대화상자를 통해 실수를 방지한다.

## 4. 디자인 가이드라인
- 대시보드 상단에 `+ 새 보드` 버튼을 고정하고, 모달 또는 슬라이드 패널로 보드 생성 폼을 제공한다.
- 필수 입력 요소: 보드 이름, 색상/아이콘(선택), 설명(선택). 빈 값에 대한 즉시 검증 메시지를 노출한다.
- 보드 카드에는 이름, 작성자, 최근 업데이트 시간을 표시하고, 컨텍스트 메뉴(...)에서 편집/삭제를 노출한다.
- 삭제 액션은 2단계 확인(경고 문구 + “보드 이름 입력”)을 요구하며, 완료 시 토스트 알림을 띄운다.

## 5. 프론트엔드 규격
- **페이지/컴포넌트**
  - `BoardsDashboardView`: 보드 목록을 카드 그리드로 렌더링하고 검색/필터 입력을 포함한다.
  - `BoardCreateModal`: 이름 입력 필드(최대 100자), 설명 텍스트 영역(500자), 색상 선택기를 포함한다.
  - `BoardActionsDropdown`: 편집, 아카이브, 삭제 옵션을 제공한다.
- **상태/데이터**
  - 보드 리스트 API 응답을 전역 상태(예: React Query)로 캐시하여 생성/편집 후 즉시 동기화한다.
  - 삭제 시 5초간 “실행 취소” 토스트를 노출하고, 취소 선택 시 프론트에서 낙관적 삭제를 롤백한다.
- **검증/에러 처리**
  - 필수 필드 미입력 시 저장 버튼을 비활성화하고 인라인 에러 메시지를 표시한다.
  - API 실패 시 모달 상단에 오류 배너(재시도 버튼 포함)를 보여준다.

## 6. 백엔드 규격
- **엔드포인트**
  - `POST /api/v1/boards`: 요청 본문(이름, 설명, 테마) 검증 후 보드 생성, 생성자/타임스탬프 기록.
  - `PATCH /api/v1/boards/{boardId}`: 부분 업데이트 허용, 변경 이력을 활동 로그에 기록.
  - `DELETE /api/v1/boards/{boardId}`: 소유자 또는 관리자 권한 확인 후 삭제, 일정 기간 소프트 딜리트 보관.
- **도메인 규칙**
  - 보드 이름 중복 허용 여부는 워크스페이스 내에서만 제한한다.
  - 삭제된 보드는 30일간 복구 가능하도록 상태 플래그와 삭제 일시를 저장한다.
- **검증**
  - 이름 1~100자, 설명 500자 제한.
  - 삭제 요청 시 하위 컬럼/카드가 존재해도 일괄 삭제되며, 활동 로그에 “보드 삭제” 이벤트를 남긴다.

## 7. 수용 기준
1. 새 보드를 생성하면 즉시 대시보드 목록과 최근 보드 섹션에 노출된다.
2. 편집된 내용은 페이지 새로고침 없이 반영되고 활동 로그에 기록된다.
3. 삭제 전 확인 과정을 거치지 않으면 실제 삭제가 실행되지 않는다.
4. 삭제 후 5초 이내 “실행 취소”를 누르면 보드가 복구된다.

## 8. 구현 순서
1. 백엔드: 보드 CRUD 엔드포인트 및 소프트 딜리트 로직.
2. 프론트엔드: 보드 목록 조회 + 생성 모달 > 편집/삭제 흐름.
3. UX 마감: 토스트·Undo, 삭제 확인 UX, 활동 로그 연동.
